SOP: Provisioning an Isolated User Ecosystem (Authelia + Nginx + Docker)

Objective: To create a secure, isolated subdomain ecosystem (Portal, Web App, and Web Terminal) for a specific user without giving them access to the main administrative portal.
Variables to Replace:
Whenever performing these steps, replace [newuser] with the actual username (e.g., ghostuser2).
Phase 1: Create the User's Landing Page
Nginx needs a physical HTML file to serve as the user's custom launchpad.
Create a new HTML file in the host directory that is mapped to Nginx's web root:

Bash
nano ~/gateway/nginx/www/[newuser].html

Paste in the HTML/CSS for their specific landing page (containing the links to their dedicated webapp and terminal subdomains).
Save and exit.
(Optional but recommended) Ensure Nginx has permission to read it:
Bash
chmod 644 ~/gateway/nginx/www/[newuser].html
Phase 2: Lock Down Authelia (Access Control)
We must explicitly tell Authelia to allow the user into their subdomains, and strictly ban them from everything else.
Open the Authelia configuration:
Bash
nano ~/gateway/authelia/configuration.yml
Locate the access_control section and add rules specific to the new user. Order matters! Put the specific subdomain allowances above the global wildcard block.
YAML
    # 1. Allow [newuser] into their specific domains
    - domain:
        - [newuser].141.147.1.61.nip.io
        - webapp-[newuser].141.147.1.61.nip.io
        - terminal-[newuser].141.147.1.61.nip.io
      subject: "user:[newuser]"
      policy: two_factor

    # 2. Block [newuser] from the main domain and all other subdomains
    - domain: "*.141.147.1.61.nip.io"
      subject: "user:[newuser]"
      policy: deny
Phase 3: Spin Up the Smart Web Terminal
The web terminal needs to be on the same Docker network as Nginx so they can talk securely without exposing ports to the public internet. It also needs SSH tools injected so it can auto-login to the target machine.
Find the name of the internal Nginx Docker network:
Bash
NET_NAME=$(docker inspect gateway_nginx -f '{{range $k, $v := .NetworkSettings.Networks}}{{$k}}{{end}}')
Spin up the terminal container. (Note: Update the target IP, password, and execution script ./ghost as needed for the specific user).
Bash
docker run -d \
  --name terminal-[newuser] \
  --network $NET_NAME \
  --restart unless-stopped \
  --entrypoint /bin/sh \
  tsl0922/ttyd:alpine -c "apk add --no-cache openssh-client sshpass && ttyd -W sshpass -p 'TARGET_PASSWORD' ssh -o StrictHostKeyChecking=no -t TARGET_USER@TARGET_IP './target_script'"
Phase 4: Configure Nginx Routing
Nginx needs dedicated server blocks for the new subdomains.
Open the Nginx configuration:
Bash
nano ~/gateway/nginx/conf.d/gateway.conf
Add three new server {} blocks at the bottom of the file (one for the landing page, one for the web app, and one for the terminal).
CRITICAL RULE: Every single server {} block that requires authentication must include the exact location = /authelia/api/verify { ... } engine block, or the routing will crash.
Template for the Landing Page:
Nginx
server {
    listen 443 ssl;
    server_name [newuser].141.147.1.61.nip.io;
    ssl_certificate /etc/nginx/ssl/gateway.crt;
    ssl_certificate_key /etc/nginx/ssl/gateway.key;

    # Paste the standard /authelia/api/verify block here

    location / {
        auth_request /authelia/api/verify;
        auth_request_set $target_url https://141.147.1.61.nip.io/authelia/?rd=$scheme://$http_host$request_uri;
        error_page 401 =302 $target_url;

        root /usr/share/nginx/html;
        try_files /[newuser].html =404;
    }
}
Template for the Terminal:
Nginx
server {
    listen 443 ssl;
    server_name terminal-[newuser].141.147.1.61.nip.io;
    ssl_certificate /etc/nginx/ssl/gateway.crt;
    ssl_certificate_key /etc/nginx/ssl/gateway.key;

    # Paste the standard /authelia/api/verify block here

    location / {
        auth_request /authelia/api/verify;
        auth_request_set $target_url https://141.147.1.61.nip.io/authelia/?rd=$scheme://$http_host$request_uri;
        error_page 401 =302 $target_url;

        # Point exactly to the container name created in Phase 3
        proxy_pass http://terminal-[newuser]:7681;
        
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $proxy_host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
(Repeat a similar block for webapp-[newuser], pointing the proxy_pass to the target web application).
Phase 5: Apply, Flush, and Test
Do not just reload Nginx. A hard restart is required to flush Nginx's internal DNS cache so it can "see" the new terminal container.
Restart Authelia to apply access rules:
Bash
docker restart gateway_authelia
Test Nginx syntax to ensure no missing brackets:
Bash
docker exec gateway_nginx nginx -t
Hard Restart Nginx to flush the DNS cache:
Bash
docker restart gateway_nginx
Final Test: Open an Incognito browser window, navigate to https://[newuser].141.147.1.61.nip.io, authenticate via Authelia, and verify all routing and terminal connections succeed.
